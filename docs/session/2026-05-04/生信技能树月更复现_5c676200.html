**需要改哪些 cell**
- 第11个代码 cell：把现有的单一模块字典改成三套输入，分别是 identity_markers、immunoreg_markers、functional_markers，同时输出每套 marker 的可用基因统计。这里是整个重构的入口。
- 第12个代码 cell：分别对三套 marker 打分，并生成 cluster-level score 汇总表。后面的命名逻辑应只依赖这些 cluster-level 分数，不再依赖 predicted_celltype 或 tissue 组成来决定名称。
- 第17个代码 cell：重写 cluster_annotation_helper。保留 predicted label、tissue、dataset 这些列做质控，但真正的命名字段只能来自三组 marker 的 top score、top2 score、margin、marker 覆盖数和置信度。
- 第18个代码 cell：删除按 cluster 编号手填的注释映射，改成从 helper 表自动生成 myeloid_subtype、myeloid_subtype_source 以及后续统一命名列。
- 第23和第24个 cell：当前是围绕 HS_UPR 做人工合并核对，这与“完全基于三组 marker 自动注释”不一致。要么删掉，要么改成“低置信度 cluster 复核”区块，不能再保留写死 subtype 的人工规则。
- 第10、16、19、21、23个 markdown cell 建议同步改说明，明确 DEG 和组成分析只是验证信息，不参与命名。

**如何严格命名 cluster，避免手工硬编码**
- 命名优先级固定为：identity 决定主谱系名，immunoreg 决定免疫调节修饰，functional 决定功能修饰。这样不会把功能状态误当成细胞身份。
- 每一类 marker 都先在 cell level 打分，再按 cluster 求均值；最好在各自类别内部做标准化，这样同一类别内的标签可比较。
- 对每个 cluster、每一类 marker 都提取 top1、top2、score margin、present marker 数、present ratio。只有满足“marker 覆盖足够 + top1 超过阈值 + top1 与 top2 差值足够大”时，才接受这个标签。
- 最终名称用固定语法自动拼接，而不是写 cluster 编号映射。推荐采用“主谱系 + 免疫调节修饰 + 功能修饰”的层级式命名；后两类若不达阈值就省略；如果 identity 本身也不稳，就标成 Unresolved 或 IdentityA_or_IdentityB，而不是人工猜测。
- predicted_celltype、celltype、tissue_type、dataset 只保留为 sanity check，不进入命名规则。这样才能真正做到完全基于用户给的 marker。
- 如果后面还要接统一命名映射，必须先固定自动标签的语法模板，否则映射文件会因为标签字符串变化而失效。

**重跑时哪些 cell 必须从头执行**
- 最稳妥的做法是从第2个代码 cell 开始顺序重跑到最后一个保留的代码 cell。因为第18个代码 cell 会重写核心注释列，后面的趋势分析和导出都依赖它。
- 如果只是最小依赖重跑，且内核没有重启，至少要重跑第11、12、17、18、20、22个代码 cell；如果你保留第24个代码 cell，也要一起重跑。
- 如果第17个代码 cell 仍然引用 cluster 组成表做质控，而中途重启过内核，还要补跑第15个代码 cell；如果 adata_mye 也不存在，则要从第2、4、6、8个代码 cell 开始重建对象。
- 第14个代码 cell 的 DEG 可以保留为验证，不是命名必需步骤；只要你的 helper 不再引用 DEG，它就不是每次必跑项。

**常见风险**
- 类别重复：同一个生物学词同时出现在 identity 和 functional，最后容易得到重复或互相覆盖的标签，需要做去重和优先级控制。
- marker 缺失：基因符号大小写、别名、Ensembl ID、过滤后缺失，都会导致某类 score 偏低，必须在第11个代码 cell 先做 availability 检查。
- 命名歧义：top1 和 top2 很接近时，直接取最大值会误标，必须用 margin 和 confidence，而不是只看第一名。
- 小 cluster 不稳定：细胞数很少时 cluster mean 容易受噪声影响，helper 表里应把 n_cells 和 low_confidence 标出来。
- 下游兼容性：第18个代码 cell 后面的统一命名映射，以及第20、22个代码 cell 的统计图，默认依赖当前字段名和标签语法，自动命名改了以后可能要同步调整。
- 隐性人工规则残留：像第24个代码 cell 这种针对特定 subtype 的专门合并逻辑，如果不清掉，最后会变成“表面自动、实际仍靠人工修补”。
